# 哔哩哔哩关注列表批量管理工具

一个功能强大的哔哩哔哩关注列表管理工具，支持批量操作、数据可视化和智能分类。

## 🎯 项目概述

## 安装与使用

### 环境要求

- Python 3.8 或更高版本
- Chrome/Edge 浏览器 (推荐)

### 快速开始

1. **克隆项目**
   ```bash
   git clone https://github.com/your-username/Bilibili-batch-operation-tool.git
   cd Bilibili-batch-operation-tool
   ```

2. **安装依赖**
   ```bash
   pip install -r requirements.txt
   ```

3. **配置Cookie**
   - 登录哔哩哔哩网页版
   - 获取Cookie信息
   - 在 `config/config.json` 中配置

4. **启动应用**
   ```bash
   # Windows
   start.bat
   
   # Linux/macOS
   ./start.sh
   ```

5. **访问界面**
   打开浏览器访问: http://localhost:8080
   
   > 💡 **端口说明**: 如果8080端口被占用，系统会自动尝试8081、8082等端口。请查看启动日志中的实际端口号。

## 功能说明

### 1. 一键更新系统 🚀

系统提供两种一键更新模式，以满足不同用户的需求和网络环境：

#### 🏃 标准模式（推荐日常使用）
**特点：**
- **API调用延迟**: 3-5秒随机延迟
- **错误容忍度**: 连续失败10次后暂停10秒
- **处理速度**: 较快，适合关注数量适中的用户
- **适用场景**: 日常维护更新，关注数量 < 500人

**功能包含：**
- ✅ 同步关注列表和分组信息
- ✅ 获取用户真实统计数据（粉丝数、视频数、播放量等）
- ✅ 修复用户等级信息  
- ✅ 智能自动分类

#### 🛡️ 保守模式（推荐首次使用）
**特点：**
- **API调用延迟**: 5-8秒随机延迟（更安全）
- **错误容忍度**: 连续失败5次后暂停20秒（更敏感）
- **智能跳过**: 跳过已有真实数据的用户，避免重复请求
- **详细日志**: 记录每个用户的同步成功详情
- **适用场景**: 首次使用、关注数量 > 500人、网络不稳定、避免风控

**额外优势：**
- ✅ 降低被B站风控的风险
- ✅ 更适合大量用户的批量处理
- ✅ 提供更详细的执行日志
- ✅ 支持断点续传（跳过已处理用户）

#### 🎯 使用建议
| 场景 | 推荐模式 | 原因 |
|------|----------|------|
| **首次使用** | 保守模式 | 降低风控风险，确保数据完整性 |
| **关注数 > 500** | 保守模式 | 大量API调用需要更谨慎 |
| **网络不稳定** | 保守模式 | 更长延迟适应网络波动 |
| **日常维护** | 标准模式 | 快速获取增量更新 |
| **遇到API限制** | 保守模式 | 更温和的请求策略 |

#### 🚀 使用方法
**Web界面操作：**
1. 访问首页：http://127.0.0.1:8080 （或实际运行端口）
2. 点击 **"一键更新所有信息"**
3. 在弹出的模态框中选择：
   - **标准模式**：常规速度，平衡性能
   - **保守模式**：更安全，降低风控风险 ⭐**推荐**

**API直接调用：**
```bash
# 标准模式
curl -X POST http://127.0.0.1:8080/api/bilibili/one-click-update

# 保守模式 (推荐)
curl -X POST http://127.0.0.1:8080/api/bilibili/one-click-update-conservative

# 注意：端口号请根据实际启动日志调整（可能是8081、8082等）
```

#### 活跃度评估原理 ✅
系统通过多维度指标综合计算UP主活跃度，帮助识别真正活跃的内容创作者：

**活跃度评估算法：** ✅ *已实现*
- **数据源**: 粉丝数量、视频总数、最后发布时间、总播放量等B站公开数据 ✅
- **时间因子**: 最后发布时间权重最高，7天内(0.9分)→30天内(0.7分)→90天内(0.5分)→更久(0.3分) ✅
- **内容产出**: 视频数量调节，100+视频(+0.1分)，少于10个视频(-0.1分) ✅
- **分数范围**: 0-1之间的标准化分数，便于比较和排序 ✅
- **默认策略**: 缺失数据时使用保守估值(0.6分)，避免误判为不活跃 ✅

**不活跃用户识别标准：** ✅ *已实现*
1. **主要标准**: 超过30天未发布新视频（基于最后发布时间）✅
2. **辅助标准**: 活跃度分数低于0.2 且 视频总数少于3个 ✅
3. **特殊情况**: 无视频活动记录且活跃度极低（<0.1）✅

**活跃度分级：** ⚠️ *部分实现*
- **高活跃(0.7-1.0)**: 近期有持续更新，内容丰富 ⚠️ *算法中未明确分级*
- **中活跃(0.4-0.7)**: 更新频率适中，有一定内容积累 ⚠️ *算法中未明确分级*
- **低活跃(0.2-0.4)**: 更新较少或间隔较长 ⚠️ *算法中未明确分级*
- **不活跃(0-0.2)**: 长期无更新或内容极少 ✅ *仅此级别在代码中有明确标准*

**不活跃用户识别机制：** ✅ *已实现*

**清理建议生成逻辑：** ✅ *已实现*

- **优先级排序**: 按不活跃程度综合评分，时间越久、活跃度越低、内容越少的排在前面 ✅

**清理建议分类：** ⚠️ *基础实现*

- **内容维度**: 内容产出极少、重复内容过多 ⚠️ *部分实现*

## ⚠️ 常见问题解决

### Q: 数据分析页面显示"不活跃用户为0"，UP主统计数据显示"无记录"？

**原因**: 数据库中只有基础的关注列表信息，缺少真实的用户统计数据（如播放量、最后视频时间等）。

**解决方案**:
1. **首次使用推荐**: 使用 **保守模式一键更新** 🏆
   - 点击首页 "一键更新所有信息"
   - 选择 "保守模式"（更安全，避免风控）
   - 等待2-3小时完成（1000+用户）

2. **或者单独同步统计数据**:
   - 首页点击 "同步数据" 按钮
   - 或访问数据分析页面点击 "同步真实数据"

3. **验证修复**:
   - 同步完成后，数据分析页面将显示真实的不活跃用户
   - UP主统计数据页面将显示完整的播放量、视频数量等信息

**说明**: 标准模式和保守模式一键更新都包含用户统计数据获取功能，区别在于API调用频率和风控防护级别。

### Q: 为什么选择保守模式？

保守模式具有以下优势：
- ✅ **更安全**: 5-8秒延迟，降低被B站风控的风险
- ✅ **智能跳过**: 避免重复获取已有数据的用户
- ✅ **适合大量用户**: 专为500+关注用户设计
- ✅ **详细日志**: 提供每个用户的详细处理信息
- ✅ **断点续传**: 意外中断后可跳过已处理用户

### Q: 服务启动后无法访问8080端口？

**解决方案**: 
- 查看启动日志，服务会自动尝试8081、8082等可用端口
- 使用日志中显示的实际端口号访问
- 示例：`http://127.0.0.1:8081`

### Q: 一键更新界面长时间没有进度更新？

**解决方案**: 正常现象，系统已优化进度显示机制
- ✅ **频繁更新**: 标准模式每处理3个用户、保守模式每处理5个用户更新一次进度
- ✅ **实时进度**: 总体进度条会随着每个用户的处理实时更新
- ✅ **详细日志**: 会显示正在处理的用户信息和处理结果
- ✅ **状态反馈**: 每个步骤都有明确的状态提示（等待中、执行中、已完成）

特别说明：用户统计数据同步阶段需要调用B站API获取每个用户的详细信息，耗时较长但界面会持续更新。

### Q: 时间估计不准确，实际耗时比预估长？

**解决方案**: 系统已根据实际情况动态计算预估时间
- 🎯 **智能估算**: 根据关注用户数量自动计算
  - 标准模式：每个用户约1.5-2秒
  - 保守模式：每个用户约2-3秒
- 📊 **实际参考**: 100个用户大约需要：
  - 标准模式：3-6分钟
  - 保守模式：5-8分钟
- ⏱️ **超时显示**: 超过60分钟时自动显示小时数
- 📱 **实时通知**: 启动时和完成时都会显示准确的预估时间

#### ⏱️ 性能优化建议

在正常网络环境下，一键更新全部信息大约需要数十分钟到数小时，主要耗时集中在「获取所有关注用户」和「逐一拉取 UP 主统计数据」两个阶段。以下是若干可行的优化思路，能够在**不牺牲最终数据完整性**的前提下缩短总体耗时：

#### 🚀 已实现的优化功能

1. **合并分组与关注列表拉取**（✅ 已实现）
   - 在优化版一键更新中，`get_follow_groups` 与 `get_all_following` 两次请求并发执行，减少一次网络往返
   - 通过 `get_following_and_groups_combined()` 方法实现，可节省约30%的初始同步时间

2. **批量获取用户统计**（✅ 已实现）
   - 使用 `asyncio.gather` 并发处理用户统计请求，将单用户延迟从 ~1s 降至 200-300ms
   - 通过 `get_user_stats_batch()` 方法实现，整体加速 3-5 倍
   - 支持可配置的并发数控制（默认8个并发）

3. **边同步边分类**（✅ 已实现）
   - 在 `_enrich_single_user()` 中集成智能分类功能，在获取用户详细信息时立即计算分类
   - 避免后续遍历时间，减少数据库操作次数

4. **增量更新策略**（✅ 已实现）
   - 为 `/one-click-update-optimized` 增加 `skip_recent_hours` 参数（默认24小时）
   - 只有当用户统计数据距离上次更新超过指定时间后才重新拉取，数据库命中率约70%
   - 智能跳过最近更新的用户，可显著减少请求数

5. **任务内复用 Cookie / 会话**（✅ 已实现）
   - 在 `BilibiliAPI` 类中使用 `aiohttp.ClientSession` 单例模式
   - 整个任务过程中复用同一个HTTP会话，避免频繁 TLS 握手与 TCP 连接建立
   - 网络开销可再下降约 10-15%
