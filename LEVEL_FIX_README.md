# UP主等级显示修复说明

## 问题描述

在关注列表和分组页面中，所有UP主的等级(LV)都显示为0，而不是真实的等级信息。

## 问题原因

B站关注列表API (`/x/relation/followings`) 返回的用户数据中不包含`level`字段，导致系统使用默认值0存储到数据库中。

## 解决方案

### 1. 配置增强

在`config/config.example.json`中添加了新的配置项：

```json
{
    "bilibili": {
        "fetch_user_details": true
    }
}
```

- `fetch_user_details`: 控制是否在同步关注列表时获取用户详细信息（包括等级）

### 2. API增强

#### 2.1 新增用户详细信息API
- `get_user_info(uid)`: 调用B站用户信息接口获取详细信息
- `_batch_enrich_user_info(users)`: 批量获取用户详细信息

#### 2.2 增强同步逻辑
- 在获取关注列表时，额外调用用户详细信息API获取等级
- 支持配置开关，可选择是否获取详细信息
- 添加错误处理和频率控制，避免API限制

### 3. 修复功能

#### 3.1 后端API
- `POST /api/bilibili/fix-user-levels`: 修复现有用户的等级信息

#### 3.2 前端界面
- 在关注列表页面和分组页面添加"修复等级信息"按钮
- 支持一键修复所有等级为0的用户

## 使用说明

### 方法一：重新同步（推荐）
1. 确保配置文件中`fetch_user_details`设置为`true`
2. 在关注列表页面点击"同步关注列表"按钮
3. 系统会自动获取用户等级信息

### 方法二：修复现有数据
1. 在关注列表或分组页面点击"修复等级信息"按钮
2. 系统会扫描所有等级为0的用户，逐一获取真实等级
3. 等待修复完成

## 注意事项

### 1. API调用频率
- 获取用户详细信息需要额外的API调用
- 系统已添加频率控制（每10个用户休息0.5秒）
- 避免频繁操作以防API限制

### 2. 性能影响
- 开启`fetch_user_details`会增加同步时间
- 建议在网络良好时进行同步操作

### 3. 错误处理
- 即使某些用户的详细信息获取失败，不会影响整体同步
- 失败的用户会保留原始数据

## 技术细节

### B站API端点
- 关注列表：`https://api.bilibili.com/x/relation/followings`
- 用户详细信息：`https://api.bilibili.com/x/web-interface/card`

### 数据库更新
- 等级信息存储在`following_list`表的`level`字段
- 修复过程直接更新数据库记录

### 前端显示
- 等级显示格式：`LV${user.level || 0}`
- 支持在详细、宫格、紧凑三种视图模式中显示

## 风控与错误处理

### 常见错误代码

#### -352 风控限制
**现象：** API返回 -352 错误代码
**原因：** 触发了B站的风控机制，可能是：
- API调用频率过快
- 短时间内大量请求
- 账号被临时限制

**解决方案：**
1. **立即停止** 所有API调用操作
2. **等待30-60分钟** 让风控自动解除
3. **检查Cookie有效性**
4. **降低操作频率**：
   - 将`api_delay`设置为2.0或更高
   - 减少并发请求
5. **分批处理**：
   - 每次只处理少量用户（如50个）
   - 处理间隔增加到数小时

#### -503 频率限制
**现象：** API返回 -503 错误代码
**原因：** API调用过于频繁
**解决方案：**
1. 增加`api_delay`配置值
2. 等待1-5分钟后重试
3. 减少批量操作的大小

#### -101/-102 登录问题
**现象：** API返回 -101 或 -102 错误代码
**原因：** Cookie失效或账号异常
**解决方案：**
1. 重新获取有效的Cookie
2. 检查账号状态
3. 确认账号未被封禁

### 风控恢复步骤

1. **识别风控**：
   ```
   2025-06-23 17:30:24 - API返回错误: -352 (code: -352)
   ```

2. **立即行动**：
   - 停止所有同步和修复操作
   - 等待至少30分钟

3. **检查配置**：
   ```json
   {
     "bilibili": {
       "api_delay": 2.0,  // 增加延迟
       "fetch_user_details": false  // 临时关闭详细信息获取
     }
   }
   ```

4. **逐步恢复**：
   - 先测试获取关注列表（不获取详细信息）
   - 确认正常后，再开启等级修复功能
   - 小批量处理（每次20-50个用户）

5. **监控日志**：
   - 密切关注API响应
   - 发现异常立即停止

### 预防措施

1. **合理配置**：
   ```json
   {
     "bilibili": {
       "api_delay": 1.5,
       "retry_times": 2,
       "fetch_user_details": true
     }
   }
   ```

2. **分时操作**：
   - 避开B站高峰时段（晚7-11点）
   - 选择凌晨或上午时段操作

3. **渐进式同步**：
   - 首次同步时关闭`fetch_user_details`
   - 基础数据同步完成后，再修复等级信息

4. **监控机制**：
   - 定期检查日志文件
   - 设置合理的错误阈值

## 故障排除

### 问题：修复后等级仍显示为0
**解决：**
1. 检查B站Cookie是否有效
2. 确认网络连接正常
3. 查看日志文件是否有错误信息

### 问题：修复过程中断
**解决：**
1. 检查是否遇到风控（-352错误）
2. 等待适当时间后重新尝试
3. 考虑分批处理

### 问题：部分用户等级无法获取
**解决：**
1. 这是正常现象，可能是用户设置了隐私保护
2. 或者该用户账号存在异常
3. 系统会记录相关日志

### 问题：连续出现API错误
**解决：**
1. 立即停止所有操作
2. 检查网络连接和Cookie
3. 等待较长时间（1-2小时）后重试
4. 考虑更换Cookie或账号

### 紧急处理流程

当遇到风控问题时：

1. **立即停止**（优先级最高）
2. **记录现状**：当前处理到第几个用户
3. **分析日志**：确认错误类型和频率
4. **等待恢复**：至少30分钟，建议1-2小时
5. **调整策略**：降低频率，分批处理
6. **逐步恢复**：从小规模测试开始
