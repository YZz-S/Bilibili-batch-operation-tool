# 智能轮询超时机制说明

## 概述

为了更好地支持大量用户的同步任务，我们移除了固定的4小时轮询超时限制，改为基于任务状态的智能检测机制。

## 新的超时检测机制

### 1. 后端超时保护
- **任务级超时**：单个同步任务最长运行6小时
- **API调用超时**：每个用户数据获取限制30秒
- **心跳机制**：定期更新任务状态，便于检测卡住的任务

### 2. 前端智能轮询
替代固定的4小时限制，新机制包含以下检测条件：

#### 立即停止条件
- **后端状态检测**：当后端标记任务为 `timeout`、`failed` 或 `stopped` 时立即停止
- **用户取消**：用户主动取消同步时立即停止

#### 智能超时检测
- **极端保护**：任务运行超过12小时强制停止（适用于超大量用户）
- **进度卡住检测**：1小时内进度无变化且状态仍为运行时判断为卡住
- **异常轮询检测**：超过7200次轮询（约10小时）时停止

### 3. 进度显示优化
- **实时耗时显示**：显示任务已运行时间
- **预估剩余时间**：基于当前进度自动计算预估完成时间
- **详细状态信息**：显示成功率、失败数量等详细信息

## 使用场景

### 小量用户（< 500人）
- 预计耗时：30分钟 - 1小时
- 轮询检测：基本不会触发超时

### 中等用户（500-2000人）
- 预计耗时：1-4小时
- 轮询检测：主要依靠进度变化检测

### 大量用户（2000-5000人）
- 预计耗时：4-10小时
- 轮询检测：12小时极端保护兜底

### 超大量用户（> 5000人）
- 预计耗时：10小时以上
- 建议：使用恢复脚本分批处理

## 恢复机制

如果同步任务被中断，可以使用恢复脚本继续：

```bash
# 从第532个用户开始恢复（适用于之前中断的情况）
python sync_resume.py 532 0

# 从头开始，只处理500个用户
python sync_resume.py 0 500

# 交互式输入参数
python sync_resume.py
```

## 监控建议

1. **观察进度变化**：正常情况下每5-6秒应该有进度更新
2. **关注成功率**：成功率持续低于30%可能遇到API限制
3. **预估时间参考**：根据显示的预估时间合理安排
4. **适时分批处理**：超过5000用户建议分批次处理

## 故障排除

### 轮询意外停止
- 检查网络连接是否稳定
- 查看浏览器控制台是否有错误信息
- 使用恢复脚本从中断位置继续

### 进度长时间无变化
- 可能遇到API频率限制，系统会自动处理
- 观察1小时后如无变化，系统会自动标记为超时
- 可手动取消后使用恢复脚本继续

### 预估时间不准确
- 预估时间基于当前进度计算，会随着处理速度变化
- API限制或网络状况会影响处理速度
- 仅供参考，实际时间可能有较大差异

## 技术细节

### 检测频率
- 轮询间隔：5秒
- 进度变化检查：每次轮询
- 超时检测：基于实际状态而非固定时间

### 数据持久化
- 每个用户处理完立即提交到数据库
- 即使任务中断，已处理的数据不会丢失
- 恢复时会跳过已有数据的用户

### 性能优化
- 后端延迟：6秒基础延迟，根据失败情况动态调整
- 内存管理：每10个用户执行一次垃圾回收
- 连接管理：自动重试和连接清理
